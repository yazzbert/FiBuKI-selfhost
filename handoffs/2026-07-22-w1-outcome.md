W1 shipped 2026-07-22: PRs #19 (spec suites), #20 (chunk 1, server core),
#21 (chunk 2, auth-shim over the real store), #22 (chunk 3, /__auth host
mount + Better Auth as default verifier), #23 (chunk 4, client rewrite +

**301 passed / 0 expected-fail**. Each PR was CI-green and adversarially
reviewed pre-merge (verdicts MERGEABLE; fixes applied in-branch, details
in the PR comments).

Selfhost auth now works end to end with no external IdP required:
provision → HTTP sign-in → JWT (JWKS + `sid` session-liveness) → data
plane, owner-scoped, one tenant. OIDC stays pluggable (`OIDC_ISSUER`),
dev bypass unchanged, Firebase build untouched except the approved
401 shaping.

## Follow-ups carried into W2/W3 (from the PR #23 review)

- **Google/social path has zero test coverage** — most likely thing to
  silently regress on a better-auth version bump.
- **Mid-pickup reload strands a live server session** (social callback
  marker is stripped before the async pickup; user must re-login).
- **Non-invited Google users fail silently** — no selfhost equivalent of
  the Firebase build's `submitAccessRequest` / "access requested" banner.
- **Login page's GitHub button is a dead control under selfhost**
  (shared UI; needs a `FIBUKI_BACKEND` gate).
- Deferred LOWs from #21: `selfhostAuth()` promise-memo poisons on
  transient boot failure (house pattern); `listUsers` omits `pageToken`.

## Operational notes

- `FIBUKI_AUTH_SECRET` is required whenever `DATABASE_URL` is set; the
  server exits with a clear message otherwise.
- Google sign-in needs `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` **and**
  `FIBUKI_AUTH_ISSUER` (redirect URI to register:
  `${FIBUKI_AUTH_ISSUER}/__auth/callback/google`); a startup warning
  fires on the half-configured state. Social cookie pickup is
  same-origin only (one-reverse-proxy deployment).
